::: {#cuda-q-device-call-channels .section}
# CUDA-Q [`device_call`{.docutils .literal .notranslate}]{.pre} Channels[¶](#cuda-q-device-call-channels "Permalink to this heading"){.headerlink}

[`cudaq::device_call`{.docutils .literal .notranslate}]{.pre} lets a
quantum kernel invoke a classical function on a remote (or co-located)
decoder/service while the program is running. The CUDA-Q compiler lowers
each call to the realtime RPC ABI; a **device-call channel** carries the
request to the service and the response back. This page describes how
channels are selected and documents the **[`cpu_roce`{.docutils .literal
.notranslate}]{.pre}** channel (a pure-CPU RDMA transport). It
complements the [[CUDA-Q Realtime Host API]{.std
.std-doc}](host.html){.reference .internal} and [[Messaging
Protocol]{.std .std-doc}](protocol.html){.reference .internal}
documents, which describe the dispatcher and the RPC wire format the
channels carry.

::: {#the-device-call-model .section}
## The [`device_call`{.docutils .literal .notranslate}]{.pre} model[¶](#the-device-call-model "Permalink to this heading"){.headerlink}

A kernel calls a classical function through
[`cudaq::device_call`{.docutils .literal .notranslate}]{.pre}:

::: {.highlight-cpp .notranslate}
::: highlight
    extern "C" int addThem(int a, int b);

    __qpu__ int kernel(int a, int b) {
      // 0 is the device id; addThem names the remote function.
      return cudaq::device_call(0, addThem, a, b);
    }
:::
:::

When compiled with realtime lowering ([`nvq++`{.docutils .literal
.notranslate}]{.pre}` `{.docutils .literal
.notranslate}[`-frealtime-lowering`{.docutils .literal
.notranslate}]{.pre}` `{.docutils .literal
.notranslate}[`--enable-mlir`{.docutils .literal .notranslate}]{.pre}),
the call is rewritten into the realtime acquire/marshal/dispatch/release
ABI. Two compile-time facts are baked in: the **device id** (the first
argument) and the **function id** ([`fnv1a_hash("addThem")`{.docutils
.literal .notranslate}]{.pre}). The channel name is *not* compiled in.
:::

::: {#selecting-a-channel .section}
## Selecting a channel[¶](#selecting-a-channel "Permalink to this heading"){.headerlink}

The channel is chosen at runtime, on the application command line, and
forwarded into the runtime by
[`cudaq::realtime::initialize(argc,`{.docutils .literal
.notranslate}]{.pre}` `{.docutils .literal
.notranslate}[`argv)`{.docutils .literal .notranslate}]{.pre}:

::: {.highlight-bash .notranslate}
::: highlight
    ./app --cudaq-device-call=<channel-name> [channel-specific args...]
:::
:::

The runtime parses [`--cudaq-device-call`{.docutils .literal
.notranslate}]{.pre}, instantiates the channel registered under that
name, and stores it in a session keyed by **device id**. At runtime,
[`cudaq::device_call(0,`{.docutils .literal
.notranslate}]{.pre}` `{.docutils .literal
.notranslate}[`...)`{.docutils .literal .notranslate}]{.pre} resolves to
the channel registered for device id 0. The same compiled binary
therefore runs over any channel by changing only the launch flag.

Additional options:

-   [`--cudaq-device-call-slots=<N>`{.docutils .literal
    .notranslate}]{.pre}: ring-buffer slot count

-   [`--cudaq-device-call-slot-size=<bytes>`{.docutils .literal
    .notranslate}]{.pre}: per-slot size

-   [`--cudaq-device-call-timeout-ms=<ms>`{.docutils .literal
    .notranslate}]{.pre}: per-dispatch timeout

The built-in channels ([`shared-memory`{.docutils .literal
.notranslate}]{.pre}, [`host-dispatch`{.docutils .literal
.notranslate}]{.pre}, [`gpu-dispatch`{.docutils .literal
.notranslate}]{.pre}) run an in-process service and are not covered in
depth here. The remainder of this page documents the
[`cpu_roce`{.docutils .literal .notranslate}]{.pre} channel.
:::

::: {#the-cpu-roce-channel .section}
## The [`cpu_roce`{.docutils .literal .notranslate}]{.pre} channel[¶](#the-cpu-roce-channel "Permalink to this heading"){.headerlink}

[`cpu_roce`{.docutils .literal .notranslate}]{.pre} carries
[`device_call`{.docutils .literal .notranslate}]{.pre} RPCs over a
**pure-CPU [`RoCEv2`{.docutils .literal .notranslate}]{.pre} RDMA**
transport, landing payloads directly in host memory at microsecond
latency. It is an *external* channel: unlike the in-process channels,
its service runs in a **separate process** (or a real FPGA) reached over
the network. The transport is [`CpuRoceTransceiver`{.docutils .literal
.notranslate}]{.pre}, described in [[CPU RoCE Transport]{.std
.std-doc}](cpu_transport.html){.reference .internal}.

::: {#wire-pattern-fpga-compatible .section}
### Wire pattern (FPGA-compatible)[¶](#wire-pattern-fpga-compatible "Permalink to this heading"){.headerlink}

The channel is the **caller** and plays the FPGA role on the wire; the
service plays the bridge/decoder role:

::: {.highlight-text .notranslate}
::: highlight
    caller (channel) --IBV_WR_RDMA_WRITE_WITH_IMM--> service   (request)
    caller (channel) <--IBV_WR_SEND--                service   (response)
:::
:::

This mirrors a real FPGA decode loop: the FPGA RDMA-Writes its syndromes
into the decoder's ring and receives the corrections as Sends. A real
HSB-enabled FPGA could therefore replace the software channel unchanged.
:::

::: {#connection-setup .section}
### Connection setup[¶](#connection-setup "Permalink to this heading"){.headerlink}

Connected (UC) queue pairs require each end to learn the other's QP
number (and, for the writer, the peer's rkey) before traffic flows. The
channel performs a minimal bidirectional rendezvous between the
transceiver's [`setup()`{.docutils .literal .notranslate}]{.pre} and
[`connect()`{.docutils .literal .notranslate}]{.pre} phases. For a real
FPGA service this exchange is driven by the HSB control plane instead;
only the rendezvous step differs, not the data plane. The runtime pulls
in no Hololink/HSB dependency.
:::

::: {#running-it .section}
### Running it[¶](#running-it "Permalink to this heading"){.headerlink}

[`cpu_roce`{.docutils .literal .notranslate}]{.pre} requires a ConnectX
NIC (or two RoCE ports / a loopback) and a service process, so it is
**not run in CI**. Provide the channel arguments as bare
[`key=value`{.docutils .literal .notranslate}]{.pre} tokens on the same
command line:

::: {.highlight-bash .notranslate}
::: highlight
    ./app --cudaq-device-call=cpu_roce \
      --cudaq-device-call-slots=64 --cudaq-device-call-slot-size=384 \
      ib-device=<channel-ib-device> local-ip=<channel-ip> \
      rendezvous-host=<service-ip> rendezvous-port=<port>
:::
:::

-   [`ib-device`{.docutils .literal .notranslate}]{.pre}: the channel's
    local IB device

-   [`local-ip`{.docutils .literal .notranslate}]{.pre}: the channel's
    RoCE [`IPv4`{.docutils .literal .notranslate}]{.pre}

-   [`rendezvous-host`{.docutils .literal .notranslate}]{.pre} /
    [`rendezvous-port`{.docutils .literal .notranslate}]{.pre}: where
    the service's rendezvous server is listening

-   the slot count/size must match the service's ring geometry
:::

::: {#test-harness .section}
### Test harness[¶](#test-harness "Permalink to this heading"){.headerlink}

The repository ships a test-only service daemon and an orchestration
script:

-   [`cpu_roce_test_daemon`{.docutils .literal .notranslate}]{.pre} runs
    the [`libcudaq-realtime`{.docutils .literal .notranslate}]{.pre}
    host dispatcher with [`CUDAQ_DISPATCH_HOST_CALL`{.docutils .literal
    .notranslate}]{.pre} handlers (keyed by [`fnv1a_hash`{.docutils
    .literal .notranslate}]{.pre}) and a TCP rendezvous server. It
    stands in for a real decoder/bridge.

-   [`cpu_roce_device_call_test.sh`{.docutils .literal
    .notranslate}]{.pre} configures the two RoCE ports and runs either:

    -   the default GoogleTest fixture ([`CpuRoceDispatchTest`{.docutils
        .literal .notranslate}]{.pre}), which drives the channel ABI
        directly, or

    -   [`--app`{.docutils .literal .notranslate}]{.pre} mode, which
        [`nvq++`{.docutils .literal .notranslate}]{.pre}-compiles a real
        [`__qpu__`{.docutils .literal .notranslate}]{.pre}
        [`device_call`{.docutils .literal .notranslate}]{.pre}
        application, spawns the daemon, runs the app over
        [`cpu_roce`{.docutils .literal .notranslate}]{.pre}, and
        verifies the compiler-lowered result.

::: {.highlight-bash .notranslate}
::: highlight
    # Compiler-driven proof over a two-port loopback:
    cpu_roce_device_call_test.sh \
      --channel-device <ibdevA> --channel-ip 10.0.0.1 \
      --daemon-device  <ibdevB> --daemon-ip  10.0.0.2 \
      --setup-network --app
:::
